## Overview

### Purpose
Provide a `/role-cr` slash command that loads the CR (Creator) meta-role and format specification into the session, enabling structured role file creation with mechanical validation. The validation script (`validate-role-output.sh`) enforces format rules that the LLM cannot bypass, catching placeholders, missing structure, permission-phrased constraints, and non-existent file paths.

### Use Cases
- User invokes `/role-cr` to create a new role file for a project session (e.g., CA, CP, CI)
- CR generates a role file following the format spec; validation catches violations before the user approves
- Validation script can be run standalone on any role file for CI/CD or manual checks

### Context
Issue 007 design decisions (from `memory/role-format-redesign.md` and MEMORY.md):
- Templates eliminated in favor of section menu model (Core/Optional/Custom)
- CR is role-agnostic — it creates any role type
- `/role-init` was eliminated; `/role-cr` replaces it as an inline skill
- Reference files already exist: `skills/role-init/reference/role-format.md` (v2.1) and `skills/role-init/reference/cr-role-creator.md` (v2)
- Validation follows the `validate-plan-output.sh` pattern (file path arg, stderr messages, non-zero exit on failure)

## Implementation Details

### Architectural Approach
Two independent deliverables that compose at the skill level:

1. **`scripts/validate-role-output.sh`** — standalone validation script. Accepts a file path and optional `--base-dir` for path resolution. Returns 0 on valid, non-zero with stderr diagnostics on invalid. Built incrementally across 4 slices (frontmatter → structure → content → paths).

2. **`skills/role-cr/SKILL.md`** — inline skill (runs in main thread). Uses DCI (`` !`cmd` ``) to inject CR role + format spec. Body contains step-by-step instructions including validation invocation and approval gate.

### Design Patterns
- **Standalone validator pattern**: Same as `validate-plan-output.sh` — file path arg, stderr messages, non-zero exit. Testable in isolation with crafted temp files.
- **DCI injection pattern**: Same as `project-conventions/SKILL.md` — `` !`cat ${CLAUDE_PLUGIN_ROOT}/path/to/file` `` loads content at skill invocation time.
- **Incremental slice building**: Slices 1-4 each add a validation rule category to the same script. Slice 5 creates the skill that ties it together.

## Known Limitations / Trade-offs

### Limitations
- Path existence checking is best-effort: it validates paths found in backtick literals and bare references, but cannot catch all possible path formats in free-form markdown
- Constraint consequence detection uses heuristic (non-whitespace text after constraint sentence), not semantic understanding
- Glob patterns (`*`, `?`, `[`) are excluded from path checks — a path that looks like a glob but isn't would be silently skipped

### Trade-offs Made
- **Inline skill vs agent**: Chose inline to keep CR in the user's main thread (interactive approval flow). Agent would fork context, losing the ability to iterate on the role file with the user.
- **`--base-dir` arg vs env var**: Chose CLI arg for explicitness and testability. CWD default covers the common case.
- **Incremental slices vs monolithic**: 4 slices for the validator adds planning overhead but ensures each rule category is independently tested before composition.

## Implementation Notes

### Key Decisions
- **DCI path**: `` !`cat ${CLAUDE_PLUGIN_ROOT}/skills/role-init/reference/cr-role-creator.md` `` — reference files stay in their existing location, no duplication
- **`--base-dir` for path resolution**: Script accepts `--base-dir <path>` (default: CWD) so the skill can pass the project root explicitly
- **Directory references**: Paths ending in `/` are checked as directories (e.g., `lib/` → `test -d`)
- **Glob exclusion**: Paths containing `*`, `?`, or `[` are skipped — these are patterns, not literal paths
- **Code block exclusion**: Placeholder detection (`{...}`, TODO, TBD) skips content inside fenced code blocks (triple backtick)
- **Constraint detection**: Lines starting with `**Never**`, `**Do not**`, or `**Only**` in the Constraints section; consequence = non-whitespace text after the constraint sentence
