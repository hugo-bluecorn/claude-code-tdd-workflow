## Overview

### Purpose
Two regression bug-fixes surfaced while dogfooding a Flutter language pack on a real
project. Both are "synthetic-fixture-passes / real-dev-bind-breaks" defects — the kind
green tests hid until a pack was exercised end-to-end. They ship together as a PATCH
(2.8.1) since both are bug-fixes with no new capability. Full write-ups: `issues/015`
(BF-001) and `issues/016` (BF-003).

### Use Cases
- A pack author dev-binds their language pack and expects no false "no convention pack"
  warning when the pack is actually resolving (BF-001).
- A Flutter project's release run bumps `pubspec.yaml` without silently destroying its
  `+build` number (BF-003).
- Any direct `bump-version.sh <bare-semver>` against a Flutter pubspec preserves the build.

### Context
Both defects are downstream of R1's Wave-1 unification onto `active-pack.sh` as the single
committed-binding resolver. BF-001 is an incomplete-rewiring: the C3 coverage check was
left on the old fetch-cache scan while the T1/projectFiles block in the same file was moved
to `active-pack.sh`. BF-003 is the bare-path `.yaml` heuristic being a faithful full-line
writer with no Flutter-build awareness — correct output on the live run depended on the
releaser model re-supplying the build, not on codified behavior.

## Requirements Analysis

### Functional Requirements
1. BF-001: the C3 no-pack coverage check recognizes dev-bound packs (it must consult the
   unified resolver `active-pack.sh "."`, not the fetch cache).
2. BF-001: a project with no binding still emits the advisory (the floor is not silenced).
3. BF-003: a bare supplied semver preserves an existing `+build` in a pubspec `version:` line.
4. BF-003: an explicit `+build` in the supplied argument overrides the file's build.
5. BF-003: a file with no build gains no spurious build (unchanged behavior).

### Non-Functional Requirements
- Full suite floor holds at 0 failed / 0 risky (bashunit 0.36.0).
- Changed scripts stay `shellcheck -S warning` clean.
- PRIME DIRECTIVE: no `tdd-*`/hook/script may reference or require a role file.

### Integration Points
- BF-001 shares the resolver `scripts/active-pack.sh` with the 7 other R1 consumers; the
  fix removes the last divergent resolver in `fetch-conventions.sh`.
- BF-003 is confined to `bump_bare_path`'s `*.yaml | *.yml` branch; the pack-driven
  `versionFiles` resolution, the `.json`/`.toml` heuristics, the object-form path, and the
  `.claude-plugin/plugin.json` self-host are untouched.

## Implementation Details

### Architectural Approach
- **BF-001:** replace the `resolved_packs[]` find-loop + `resolve-active-pack.sh`
  invocation (lines ~163-174) with a single
  `active_packs="$(bash "${hook_dir}/../scripts/active-pack.sh" "." 2>/dev/null)"`.
  The downstream `read_pack_for_c3` / `c3_marker_covered` logic is unchanged —
  `active-pack.sh` already emits one pack dir per line, which `c3_marker_covered`
  iterates. This collapses two resolvers into one and is also more correct: it asks
  "what packs are active for THIS project," not "what packs exist anywhere in the cache."
- **BF-003:** in the `*.yaml | *.yml` case, branch on whether the supplied `$VERSION`
  contains a `+`. If it does, write it verbatim (explicit build wins). If it does not,
  preserve an existing `+build` from the file via a capture group — substitute only the
  semver portion, leaving any `+...` suffix intact.

### Design Patterns
- **Single source of truth (BF-001):** one resolver (`active-pack.sh`) for all
  binding/active-pack questions — kills the divergence permanently rather than patching
  the same trap twice (same lesson as H1's shared `iterate-binding.sh` split).
- **Caller-owns-the-build, but don't destroy (BF-003):** the caller still owns the build
  when it supplies one; the fix only stops the heuristic from destroying a build the
  caller omitted.

### File Structure
```
hooks/fetch-conventions.sh                              # BF-001 fix (C3 block)
test/hooks/fetch_conventions_no_pack_warn_test.sh       # BF-001 tests (extend)
scripts/bump-version.sh                                 # BF-003 fix (bump_bare_path yaml case)

```

## Known Limitations / Trade-offs

### Limitations
- BF-001's C3 advisory remains exactly that — advisory stderr; it never blocks. The fix
  corrects the false-negative, not the advisory's non-blocking nature (by design).
- BF-003 fix-option (a) addresses the bare-path `.yaml` heuristic only. The object-form
  `{path,pattern}` entry already lets a pack author write an exact pattern, so it needs no
  build-awareness; `.json`/`.toml` files have no Flutter `+build` concept.

### Trade-offs Made
- BF-003: chose option (a) — preserve-in-bump-version — over (b) codify-in-releaser or
  (c) document-the-contract. Rationale: (a) makes the script correct for ANY caller
  (including a direct CLI invocation), not just the careful releaser; it is the smallest
  change that closes the latent risk at the source.

## Implementation Notes

### Key Decisions
- **BF-001 mirrors the T1 block:** rather than invent a new resolution path, reuse the
  exact `bash active-pack.sh "."` idiom already proven correct two sections below in the
  same file.
- **BF-003 branches on `+` in the argument:** an explicit build in the supplied version is
  honored verbatim; absence of a build triggers preservation of the file's build.

### Future Improvements
- A broader sweep could audit the remaining deferred issue-014 latent items, but those are
  out of scope for this PATCH.

### Potential Restructuring
- If a third resolver consumer ever appears, the `active-pack.sh "."` call could be wrapped
  in a tiny helper — left for the implementer to judge at implementation time; not planned.
